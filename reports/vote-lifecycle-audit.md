# Vote lifecycle audit

`ProcessorApp` registers the tools (`processor/src/main/java/ee/ivxv/processor/ProcessorApp.java:35-47`). `RevokeTool.run` loads each revocation list and invokes `bb.revokeDoubleVotes` (`RevokeTool.java:70-87`); `RevokeAndAnonymizeTool.run` performs the same operation, writes revocation reports, then calls `anonymize` (`RevokeAndAnonymizeTool.java:59-95`). The callback records voter IDs and ballot references (`191-203`) before the anonymous box is emitted. `AnonymizeTool.run` independently calls `BallotBox.anonymize` (`AnonymizeTool.java:24-64`).

These call paths establish the stage ordering, but the exact comparator for latest qualified ballot, tie handling, and malformed timestamp behavior is in `BallotBox`/qualification model code and requires line-by-line fixture tests. The source exposes an error message for equal latest times (`processor/Msg.java:64`), indicating ties are rejected rather than silently ordered; this is a source observation, not an end-to-end proof. Count-conservation, contradictory revoke/restore lists, reordered inputs, and identity-field absence remain untested in the fixed publication. Invariants are recorded in [vote-lifecycle-invariants.tsv](../evidence/source-audit/vote-lifecycle-invariants.tsv).
