# Collector and voting audit

This focused workstream reviewed exact files: `common/collector/zip/zip.go:86-143,231-270,504-643` and `zip_test.go:148-430`; `common/collector/safereader/reader.go:10-58` and `reader_test.go:17-108`; `common/collector/auth/auth.go:88-161`; `common/collector/q11n/ocsp/ocsp.go:22-61`; `common/collector/q11n/tsp/tsp.go:38-100`; and storage interfaces `common/collector/storage/storage.go:97-146,631-753,1224-1258`. The complete network service path remains in progress: `common/collector/server`, identity backends and application-specific handlers are not marked reviewed.

The qualification path is registered in `q11n/ocsp/ocsp.go:init` and `q11n/tsp/tsp.go:init`; `client.Qualify` performs the external OCSP/TSP exchange (`ocsp.go:49`, `tsp.go:83`) and returns an error on protocol failure. The collector therefore has an explicit qualification-failure path rather than silently accepting an unqualified container.

`auth.Configure` selects a registered authenticator (`auth.go:105-134`) and `Auther.Verify`/`Data` dispatch verification (`136-161`); backend failures return errors. Storage `Client.ensure`/`update` call the protocol Put and wrap failures (`storage.go:97-146`), while `SetVoted`/`CheckVoted` persist and read voter state (`631-753`). `StoreVote` validates required qualifying properties before storage (`1224-1258`). These are concrete trust-boundary/error paths, but network acknowledgement and same-voter concurrency require tracing `server` and application handlers still outstanding.

The storage boundary is explicit in `storage/service/storage/main.go:newCtrl` (`61-80`): configuration is parsed, CA material is retained in memory, and TLS is configured with a minimum of TLS 1.2. Startup writes the CA (`276-281`), creates a readiness socket (`283-290`), starts etcd (`293-307`), and forwards stdout/stderr to the logger. A failed write, socket bind, logger creation, or child-process start returns an error; no success acknowledgement is reachable before these steps. The child process is nevertheless not context-killed (comment at `293`), so timeout/stop behavior must be tested under process failure.

Replay and concurrency are not established by the available source-only fixtures: the identified-ballot key construction and write acknowledgement are in storage client code, while the online network handler is spread across service packages. Phase 5 must exercise duplicate requests, retry after timeout, concurrent same-voter submissions, and etcd restart. No production endpoint was contacted.
